# 技术支柱 (Technical Pillars)

MindOS 的长期技术壁垒不在于"功能多"，而在于定义 Agent 时代操作系统的核心组件。

## 为什么是这四个

从 MindOS 的三大痛点（记忆割裂、记忆黑箱、经验断流）出发，解决路径上必然遇到四个系统级问题：

1. Agent 无法精确执行人类的非结构化知识 → **双向编译**
2. 知识库长期使用后质量退化 → **记忆代谢**
3. 多 Agent 并发读写无序 → **群体调度**
4. 通用 Agent 不懂个体用户 → **认知镜像**

"安全与隐私"是贯穿所有 Pillar 的横切关注点（本地优先、路径沙箱、写保护已内嵌于架构），不单独成柱。

## 结构关系

**三个独立基础设施 + 一个顶层涌现。**

```
Pillar 1 (双向编译) ──┐
                      │
Pillar 2 (记忆代谢) ──┼──→ Pillar 4 (认知镜像)
                      │
Pillar 3 (群体调度) ──┘
```

Pillar 1/2/3 相互独立、可并行推进，各自解决不同的系统层问题。Pillar 4 是在三者之上的涌现能力——当系统能精确执行指令、能自主管理记忆、能协调多 Agent 后，自然沉淀出对用户的深度理解。

## Pillar 1 — 人机双向编译与共生演化

**类比：** MindOS 的编译器。

**核心问题：** 人类习惯散漫的自然语言，Agent 需要结构化的可执行指令——两者之间缺少一个双向翻译层。

**当前基础：**
- MindOS Skills 让 Agent 读取 Markdown 格式的 SKILL.md 并按其指引执行，但本质是 prompt 注入——LLM 直接读原文，没有解析为中间表示，离真正的"编译"有很大距离
- `experience.md` 经验回流目前是人工编写，系统没有自动从对话中提取和结构化的能力
- Skill 的 SKILL.md 描述 + 执行模式是半结构化的行为定义，可作为未来中间表示的起点

**Gap 与研究方向：**

| 子问题 | 说明 |
|--------|------|
| 价值验证 | 中间表示比 prompt 注入好在哪？需先找到 LLM 直读 Markdown 表现差的场景（多步条件分支、跨文件依赖、执行顺序敏感），用这些 bad case 证明编译的必要性 |
| 结构化表示 | 将 Markdown SOP 解析为可执行的中间表示（行为树 / DAG / 状态机），而非让 LLM 每次重新理解原文 |
| 反向编译 | Agent 执行出错 → 人类介入修正 → 系统自动从交互增量中提取 diff → 更新底层结构 → 反映回 Markdown 供审计。注意：处理的是**实时交互产生的增量经验**（对话→SOP），区别于 Pillar 2 的静态文件去重 |
| 增量演化 | 不是全量重写，而是在已有结构上做拓扑级别的局部修改（插入分支、修改条件、归档旧路径） |

**与 Roadmap 的对应：** 独立研究线。P2 的 Skills 增强是前置工程准备，核心算法落地预计 P3+。

## Pillar 2 — 动态记忆代谢与自组织心智网络

**类比：** MindOS 的文件系统——让知识库从"死水"变成有生命周期的活体。

**核心问题：** 知识库只增不减，Agent 越用越迟钝。RAG 检索出一堆过期的、矛盾的、重复的内容（上下文污染）。

**当前基础：**
- Knowledge Health（P2 规划）已有过期检测、孤立文件、矛盾扫描的设想，但尚未实现
- Wiki Graph 提供了文件间链接的拓扑结构
- Agent Inspector 记录了工具调用日志，具备"谁在用什么"的原始数据

**Gap 与研究方向：**

| 子问题 | 说明 |
|--------|------|
| 记忆分层 | 工作记忆（当前对话上下文）→ 短期记忆（近期交互沉淀）→ 长期记忆（全局 SOP / Profile），三层之间有升维和降级机制 |
| 活跃度感知 | 基于 Agent 调用频率、用户编辑频率、引用关系，维护每个知识节点的"温度"——高温节点预热，低温节点归档 |
| 碎片折叠 | 当**已有的静态文件**中多条记录语义高度重叠时，自动触发合并提炼，生成更高层级的抽象节点。区别于 Pillar 1 的反向编译（处理实时交互增量） |
| 矛盾检测与解决 | 同一主题的不同文件出现矛盾描述时，主动标记冲突并引导用户裁决 |

**与 Roadmap 的对应：** P2（Knowledge Health 是工程前置）、P3（深度 RAG、Agent Memory Layer）

## Pillar 3 — 异构群体智能调度与治理

**类比：** MindOS 的 OS 内核——不只提供数据，更负责多 Agent 的进程管理。

**核心问题：** 用户同时接入 5-10 个异构 Agent（写代码的、查资料的、管日程的），它们并发读写同一个知识库，谁来维持秩序？

**当前基础：**
- MCP Server 已支持多 Agent 同时连接（HTTP + stdio 双传输）
- Bearer Token 认证提供了基础的身份识别
- Agent Governance（P3 规划）已有 ACL、配额、审批流的设想
- 用户已经在同时使用 Claude Code + Cursor 等多 Agent 场景，并发问题**现在就存在**

**Gap 与研究方向：**

| 子问题 | 说明 |
|--------|------|
| 任务感知的上下文路由 | 根据 Agent 当前任务类型和关键性，动态裁剪返回的上下文——重构代码的拿完整架构文档，拼写检查的只拿当前段落，而不是让每个 Agent 自己拼 prompt |
| 并发冲突解决 | 两个 Agent 同时修改同一文件时的锁机制、乐观并发控制、冲突合并策略 |
| 细粒度权限 | 不是简单的读/写二分，而是"Claude Code 可写代码区但只读 Profile""Cursor 可读架构文档但不能碰 SOP" |

**应用层扩展：**
- **多 Skill 编排** — 多个 Skill 的执行逻辑如何组合、编排、检测冲突；多 Agent 协作完成复合任务时的消息传递和结果聚合。建立在上述三个基础设施能力之上。

**与 Roadmap 的对应：** P3（Agent Governance、多 Agent 协作中枢、ACP 协议）。并发冲突解决的基础工程可在 P2 提前启动。

## Pillar 4 — 个性化认知镜像与隐性意图对齐

**类比：** MindOS 的用户态——工具可以被复制，但"懂你"的程度无法被复制。

**核心问题：** 通用 Agent 对所有用户一视同仁。MindOS 掌握用户最深度的交互数据，如何把这种理解转化为不可替代的体验？

**当前基础：**
- Profile 目录已存储用户的身份、偏好、目标、风格
- Agent Inspector 记录了完整的交互轨迹
- 经验回流机制（人工编写）积累了用户的决策模式样本

**与通用推荐系统的关键区别：** MindOS 的数据源不是点击流或浏览行为，而是**纯文本交互日志 + 用户自主编写的知识库**。本地优先意味着没有大规模协同过滤的数据基础，必须从单用户的长期文本轨迹中推断偏好——这本身就是一个独特的研究约束。

**Gap 与研究方向：**

| 子问题 | 说明 |
|--------|------|
| 文本轨迹偏好推断 | 从 Agent 调用日志（哪些工具被频繁使用、哪些建议被采纳/拒绝）和用户编辑行为（修改了 Agent 写入的哪些内容）中，提取隐性偏好 |
| 心智预设注入 | 外部 Agent 通过 MCP 请求上下文时，MindOS 在响应中附带一段结构化的用户特征摘要（如 `_meta.user_profile` 字段）。Agent 可选择性读取并注入自己的 system prompt。初期可从 Profile 目录静态生成，后期从交互轨迹动态更新 |
| 意图预测 | 基于用户当前的工作上下文和历史模式，主动推荐可能需要的知识节点，而不是等用户搜索 |
| 演化追踪 | 记录用户认知模型随时间的变化轨迹（Mind Diff），让用户能回溯"我三个月前是怎么想这个问题的" |

**与 Roadmap 的对应：** P4（长期探索）。数据采集基础可在 P2-P3 的 Agent Inspector 增强中积累。

## 与竞品的差异化

| 能力 | Obsidian + AI | Notion AI | Mem.ai | Khoj | **MindOS** |
|------|--------------|-----------|--------|------|-----------|
| 多 Agent 统一记忆 | 插件拼装 | 否 | API 接入，非 MCP 原生 | 单入口 | **MCP 协议原生支持** ✅ |
| 本地优先 | 是 | 否 | 否 | 是 | **是** ✅ |
| 开源 | 核心闭源 | 否 | 否 | 是 | **是** ✅ |
| 知识可执行 | 否 | 否 | 否 | 否 | Skills + 双向编译 🔬 |
| 记忆代谢 | 手动整理 | 否 | 自动但黑箱 | 部分（自动索引） | 自动 + 透明可审计 🔬 |
| 多 Agent 治理 | 否 | 否 | 否 | 否 | ACL + 上下文路由 🔬 |
| 认知镜像 | 否 | 否 | 否 | 否 | 交互轨迹 + 意图预测 🔬 |

> ✅ = 已实现　🔬 = 研究方向，尚未实现
