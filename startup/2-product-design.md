## 二、双层楔子策略

> 原始设计文档由 /office-hours 于 2026-03-26 生成，经 CEO Review 和战略讨论演进。

### 用户痛点

| 痛点 | 描述 | 证据来源 |
|------|------|----------|
| 上下文割裂 | 换 Agent 靠人工搬运上下文 | 创始人经验 + 内测反馈（N≈10） |
| 治理黑箱 | Agent 记了什么，用户无法审查修正 | 创始人经验，待验证 |
| 判断断流 | 纠正和判断关掉就丢，无法复用 | 创始人经验，待验证 |

### 双层策略

| 层级 | 价值 | 对应功能 | 用户感知 |
|------|------|----------|----------|
| 获客层（止痛药） | 不用重复讲背景 | MCP Server、一键导入、模板 | "AI 终于记住我了" |
| 留存层（维生素） | 判断沉淀，越用越准 | Skills、Instruction、经验回流 | "AI 越用越懂我" |

**一句话：** 用户来止痛，结果发现自己和 AI 都在变强。

### Premise Challenge 结论

1. 楔子和地基对齐（GUI/渲染器是地基）
2. 首次体验路径偏长 → 已通过一键导入+新手引导缓解
3. 证据缺口是最大风险 → 用户访谈是当前最高优先级
4. 叙事重组：先讲痛点（止痛药），认知复利作为飞轮涌现结果
5. 人机共演化不是口号，是飞轮转起来后的自然现象

## 五、验证计划

### 融资前必须完成

1. 找 10 个非朋友用户深度访谈，量化"反复讲背景"的时间成本
2. 找 3 个用户愿意付 $8/月测试 Pro 功能
3. 跟踪 MCP 协议在 OpenAI/Google 生态的采纳情况

### 观察要点

只观察三件事：

1. 用户第一步去哪里找项目记忆
2. 在哪一步开始卡住
3. 有没有自发把判断或 SOP 写回系统（共演化的早期信号）

## 六、10x Vision

MindOS 成为你和所有 AI 之间的判断沉淀层：

1. 用户正常工作，判断和偏好持续沉淀
2. 任何新 Agent 接入，零配置读到你的全部规则
3. 飞轮转起来后，AI 变准，人变清晰——双方螺旋上升
4. 到那时用户不是被锁定了，而是自己变成了不同的人

### Platonic Ideal

> 你打开 MindOS，什么都不用做。它已经知道你在做什么项目、你的偏好是什么、你上次纠正了 AI 哪里。你切到 Cursor，Cursor 知道你的架构决策。切到 Claude Code，Claude 知道你的规范。
>
> 但更重要的是——你回看 3 个月前写的规则，发现自己的思维比那时清晰了 10 倍。不是 MindOS 教你的，是你自己在"给 AI 写规则"的过程中变强的。
>
> 用户的感受："不只是 AI 在变好——我也在变好。"

## 七、反哺认知功能设计（层次 4 产品化）

> 层次 1-3 让 AI 越来越懂你。层次 4 让你越来越懂你自己。这是 MindOS 的付费核心和最深护城河。

### 映（Mirror）——让你看到自己

| 功能 | 做什么 | 用户看到什么 | 阶段 |
|------|--------|------------|------|
| **纠正热力图** | 统计用户对 AI 纠正的分布 | "你 60% 的纠正关于代码风格——这是你的核心审美" | P2 |
| **认知时间线** | 把规则按时间排列，展示进化轨迹 | "你 3 个月前说'用字符串'，现在说'用枚举'。你在进化" | P2 |
| **盲区地图** | 可视化规则覆盖的维度，标出空白 | "你覆盖了编码/架构/文案 3 个维度，测试和安全是空白" | P2 |
| **决策模式画像** | 分析决策倾向和风格 | "你在技术选型上偏保守，在产品方向上偏激进" | P3 |

**产品形态：** Echo 面板中新增 "Insights" 页签。用户打开 Echo 不只看到自己写的反思，还看到 AI 基于全量数据生成的认知分析。视觉上用琥珀色卡片，和用户手写内容（白色）区分——"这些是 AI 帮你看到的，你自己看不到的"。

### 鉴（Assess）——给你评估

| 功能 | 做什么 | 用户看到什么 | 阶段 |
|------|--------|------------|------|
| **知识库健康度** | 扫描冗余率、矛盾数、过期文件数 | "冗余率 12%，矛盾 3 处，6 个月未引用文件 47 个" | P2 |
| **规则质量评分** | 评估每条规则的清晰度、可执行性、引用频率 | "规则 #17 被引用 23 次，规则 #42 从未被引用" | P2 |
| **Agent 遵守率** | 统计 Agent 实际遵守了多少规则 | "上周 Agent 遵守率 78%，违反最多的是规则 #5" | P2 |

**产品形态：** Settings 中新增 "Health" 仪表盘。一个数字（健康度评分 0-100），点击展开详情。每项有"修复"按钮——点击自动跳转到问题规则的编辑界面。每月自动生成健康报告推送到 Echo Daily。

### 澄（Clarify）——帮你想清楚

你以为自己写清楚了，其实没有。Agent 比你更容易发现模糊——因为人读到"简洁"会脑补，AI 读到"简洁"会困惑。AI 的困惑就是你的模糊。

| 功能 | 做什么 | 用户看到什么 | 阶段 |
|------|--------|------------|------|
| **模糊规则检测** | 识别含糊、不可执行的规则 | "你写了'代码要简洁'。什么叫简洁？20 行以内？变量名短？函数少？" | P2 |
| **澄清对话** | 针对模糊规则发起追问，引导用户精确化 | "你说'错误处理要友好'——友好是给用户看的还是给开发者看的？" | P2 |
| **执行歧义报告** | 统计 Agent 在执行规则时产生歧义的频率 | "规则 #8 在过去一周被 3 个 Agent 以不同方式理解" | P2 |
| **边界条件追问** | 规则在极端场景下的适用性 | "你说'架构要灵活'。如果灵活和性能冲突呢？哪个优先？" | P3 |
| **隐含假设揭示** | 找出规则背后未说出的前提条件 | "你说'所有 API 用 RESTful'——这里隐含了'不做实时推送'的假设，你确认吗？" | P3 |

**产品形态：** 两种入口。第一种是**被动提示**：规则编辑器中，模糊规则旁边出现淡蓝色问号图标，hover 显示"这条规则可能不够精确"，点击展开 AI 的追问。第二种是**主动对话**：用户点击"帮我想清楚"按钮，AI 对选中的规则发起苏格拉底式追问——不给答案，用问题逼你把模糊变精确。每次澄清的结果自动更新到规则里。

**核心价值：** 这不是纠错，是帮你思考。很多时候你不是"写错了"，而是"还没想透"。MindOS 用 AI 的困惑倒逼你的清晰。写完规则的人和写之前的人不是同一个人——他的思维被精确化了。

### 校（Calibrate）——帮你纠偏

| 功能 | 做什么 | 用户看到什么 | 阶段 |
|------|--------|------------|------|
| **规则矛盾检测** | 语义分析发现规则 A 和 B 互相矛盾 | "规则 A 说'函数不超过 20 行'，规则 B 说'一个函数做完整一件事'。你信哪个？" | P2 |
| **过时规则提醒** | 检测长期未引用的规则 | "这条规则 3 个月没被引用了。还需要吗？" | P2 |
| **路径依赖挑战** | 检测重复模式并质疑 | "你 20 个项目都用 REST。最近 3 个更适合 GraphQL。有意的？" | P3 |

**产品形态：** 规则编辑器中的 inline 提示。不弹 modal，不打断——在规则旁边用淡琥珀色边框标注"发现矛盾"或"可能过时"。用户可以忽略（标记为"有意为之"）或点击处理。矛盾解决后自动清除标注。

### 启（Enlighten）——让你升维

| 功能 | 做什么 | 用户看到什么 | 阶段 |
|------|--------|------------|------|
| **碎片→原则提炼** | 把 N 条碎片规则归纳成核心原则 | "你的 50 条规则中有 18 条在说同一件事。可以归纳为：'永远优先可读性'" | P3 |
| **跨项目模式识别** | 发现跨项目的重复决策模式 | "你在不同项目里反复做同一个决策，这可能是你的核心方法论" | P3 |
| **成长报告** | 月度/季度总结 | "本月新增 12 条规则，修改 5 条，你的规范覆盖率从 60% 提升到 75%" | P2 |

**产品形态：** 月度邮件 + Echo Growth 页签。成长报告是最强的付费触发点——Free 用户也能看到"你本月有一份成长报告"，但打开需要 Pro。报告不是冰冷的数据表，而是一段叙事："这个月你最大的变化是..."。

### 反哺认知的路线图
